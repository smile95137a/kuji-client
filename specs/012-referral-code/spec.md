## 使用者情境與測試

### 使用者故事 1 — 不使用推薦碼正常註冊（優先級：P1）

新訪客填寫 Email 與密碼，不輸入推薦碼，完成帳號建立。

**此優先級的原因**：推薦碼為選填欄位，絕大多數新用戶不會有推薦碼，主流程不可因此中斷。

**獨立測試**：
1. 進入 `/register`，填寫 Email、密碼、確認密碼。
2. 推薦碼欄位留空。
3. 點擊「立即註冊」。
4. 確認 API Request body 中不含 `referralCode` 欄位或值為 `null`。

**驗收情境**：
1. **在** 推薦碼欄位為空，**當** 玩家點擊「立即註冊」，**則** `POST /api/auth/register` Request body 不含 `referralCode`（或值為 `null`/空字串），不顯示任何錯誤。
2. **在** 註冊 API 回應成功，**當** 收到 `{ userId, email, token, refreshToken }`，**則** 自動登入（儲存 token），顯示「🎉 歡迎加入 KUJI！」Toast，導向首頁 `/`。
3. **在** Email 已被註冊，**當** API 回應 409 錯誤，**則** Email 欄位顯示「此 Email 已被使用，請改用其他 Email」錯誤訊息。

### 使用者故事 2 — 輸入有效推薦碼並完成註冊（優先級：P1）

新訪客輸入有效的店家推薦碼，系統即時驗證並在成功後顯示店家資訊，完成帶推薦碼的註冊。

**此優先級的原因**：推薦碼是重要的用戶獲取機制，正確的視覺反饋（顯示店家名稱）提升信任感與轉換率。

**獨立測試**：
1. 進入 `/register`，在推薦碼欄位輸入有效碼（如 `KUJI2026`）。
2. 等待 600ms debounce 後，確認 `POST /api/referral/validate` 被呼叫。
3. 確認 UI 顯示「✅ 已連結至 [店家名稱]」。

**驗收情境**：
1. **在** 玩家在推薦碼欄位輸入文字並停頓 600ms，**當** debounce 觸發，**則** 自動呼叫 `POST /api/referral/validate`，欄位右側顯示旋轉 Spinner（⏳）。
2. **在** 驗證 API 回應 `{ valid: true, storeName: "扭蛋王國" }`，**當** 收到，**則** 推薦碼欄位顯示綠色邊框，欄位下方顯示「✅ 已連結至 扭蛋王國」（綠色文字），Spinner 消失。
3. **在** 推薦碼有效，**當** 玩家點擊「立即註冊」，**則** Request body 包含 `referralCode: "KUJI2026"（原始輸入的代碼）。
4. **在** 帶推薦碼的註冊 API 成功，**當** 收到 Response，**則** 顯示「🎉 歡迎加入 KUJI！您已透過推薦碼加入 扭蛋王國」Toast，導向首頁。

### 使用者故事 3 — 輸入無效推薦碼（優先級：P1）

玩家輸入不存在或已停用的推薦碼，UI 即時顯示錯誤但允許繼續提交（推薦碼選填）。

**此優先級的原因**：推薦碼無效時的處理方式影響用戶體驗：若阻止提交，可能因推薦碼失效而失去用戶；若靜默忽略，用戶可能不知道碼無效。

**獨立測試**：輸入不存在的推薦碼，確認 UI 反饋後，嘗試直接提交。

**驗收情境**：
1. **在** 驗證 API 回應 `{ valid: false, message: "推薦碼無效或已過期" }`，**當** 收到，**則** 推薦碼欄位顯示紅色邊框，欄位下方顯示「❌ 推薦碼無效」（紅色文字），Spinner 消失。
2. **在** 推薦碼驗證為無效，**當** 玩家仍點擊「立即註冊」，**則** 顯示確認對話框「推薦碼無效，是否不使用推薦碼繼續？」，玩家確認後以空推薦碼提交。
3. **在** 玩家清空推薦碼欄位，**當** 欄位變為空白，**則** 驗證狀態重置（清除驗證指示器），不顯示任何驗證訊息。
4. **在** 推薦碼正在驗證中（⏳ 狀態），**當** 玩家點擊「立即註冊」，**則** 等待驗證完成後再提交，或顯示「請等待推薦碼驗證完成」提示。

### 使用者故事 4 — 透過推薦連結進入並自動填入推薦碼（優先級：P2）

玩家透過店家分享的推薦連結進入（如 `https://kuji.example.com/register?ref=KUJI2026`），推薦碼欄位自動填入並觸發驗證。

**此優先級的原因**：自動填入減少玩家操作步驟，提升推薦轉換率。

**驗收情境**：
1. **在** 玩家透過 `/register?ref=KUJI2026` 進入頁面，**當** 頁面 `onMounted`，**則** 推薦碼欄位自動填入 `KUJI2026` 並立即觸發驗證（不需 debounce，直接呼叫）。
2. **在** 自動填入驗證成功，**當** 頁面顯示，**則** 玩家看到「✅ 已連結至 [店家名稱]」，減少疑慮。
3. **在** URL 的 `ref` 參數為無效碼，**當** 頁面載入，**則** 顯示「❌ 此推薦碼無效或已過期」，欄位保留原始碼供用戶確認或清除。

### 使用者故事 5 — 表單欄位即時驗證（優先級：P1）

所有必填欄位在使用者離焦（onBlur）時即時驗證，不等到提交時才顯示錯誤。

**驗收情境**：
1. **在** Email 欄位，**當** 玩家輸入無效格式（如 `notanemail`）後離焦，**則** 立即顯示「請輸入有效的 Email 格式」。
2. **在** 密碼欄位，**當** 玩家輸入不符合規則（< 8 字元）後離焦，**則** 顯示「密碼至少需 8 個字元」。
3. **在** 確認密碼欄位，**當** 輸入值與密碼不符後離焦，**則** 顯示「密碼與確認密碼不符」。
4. **在** 所有必填欄位已填寫且無錯誤，**當** 玩家點擊「立即註冊」，**則** 執行 API 呼叫（不再阻擋）。

## API 串接規格

### 推薦碼與註冊

| 動作 | Method | 端點 | 主要 Request 欄位 | 主要 Response 欄位 | 前端 UI 狀態 |
|------|--------|------|------------------|--------------------|-------------|
| 驗證推薦碼 | POST | `/api/referral/validate` | `code: "ABC123"` | `valid`, `storeName`, `message` | 驗證指示器（✅ / ❌） |
| 使用者註冊 | POST | `/api/auth/register` | `email`, `password`, `referralCode`(optional) | `userId`, `email`, `token`, `refreshToken` | 成功 Toast + 自動登入 |

#### 驗證 Request 範例
```json
POST /api/referral/validate
{
  "code": "KUJI2026"
}
```

#### 驗證 Response（有效）
```json
{
  "valid": true,
  "storeName": "扭蛋王國",
  "message": "推薦碼有效"
}
```

#### 驗證 Response（無效）
```json
{
  "valid": false,
  "storeName": null,
  "message": "推薦碼無效或已過期"
}
```

#### 註冊 Request 範例（含推薦碼）
```json
POST /api/auth/register
{
  "email": "player@example.com",
  "password": "SecureP@ss1",
  "referralCode": "KUJI2026"
}
```

## 功能需求（前端 UI）

### 推薦碼欄位

- **FR-UI-001**：推薦碼輸入框 Placeholder 文字：「選填 — 輸入推薦碼」。
- **FR-UI-002**：欄位右側有狀態指示器區域：⏳（驗證中）/ ✅（有效）/ ❌（無效）/ 空（未輸入或已清空）。
- **FR-UI-003**：驗證成功時，欄位下方以小字顯示「✅ 已連結至 {storeName}」（綠色）。
- **FR-UI-004**：驗證失敗時，欄位下方以小字顯示「❌ 推薦碼無效」（紅色）。
- **FR-UI-005**：推薦碼輸入轉為大寫（`text-transform: uppercase`），或在 onChange 時自動 `.toUpperCase()`。
- **FR-UI-006**：驗證 API 使用 debounce 600ms（稍長於一般搜尋，因為每次驗證有後端成本）。
- **FR-UI-007**：在 debounce 觸發前若用戶清空欄位，取消任何進行中的 API 請求（AbortController）。

### 表單驗證規則

| 欄位 | 驗證規則 | 錯誤訊息 |
|------|---------|---------|
| Email | 必填，RFC 5322 格式 | 「請輸入有效的 Email 格式」 |
| 密碼 | 必填，≥ 8 字元，含英文和數字 | 「密碼至少 8 字元，需含英文與數字」 |
| 確認密碼 | 必填，需與密碼相同 | 「密碼與確認密碼不符」 |
| 推薦碼 | 選填，英文+數字，6–10 字元 | 「推薦碼格式不正確（6-10碼英數）」 |
| 服務條款 | 必須勾選 | 「請先閱讀並同意服務條款」 |

### 密碼安全顯示

- **FR-UI-008**：密碼欄位右側顯示「眼睛」圖示，點擊切換 `type="password"` 和 `type="text"`。
- **FR-UI-009**：密碼強度指示條（弱/中/強），根據密碼複雜度（長度、大小寫、特殊字元）即時更新。

### 狀態管理

- **SM-001**：`formData` — `{ email, password, confirmPassword, referralCode, agreeToTerms }`。
- **SM-002**：`formErrors` — 各欄位的錯誤訊息 Map。
- **SM-003**：`referralValidation` — `{ status: "idle"|"loading"|"valid"|"invalid", storeName: null }`。
- **SM-004**：`isSubmitting` — 控制提交按鈕禁用。
- **SM-005**：`abortController` — 用於取消正在進行的推薦碼驗證 API 請求。

## API 驗證清單

- **AV-001** ✅：`POST /api/referral/validate` 回傳 `valid`（boolean）和 `storeName`（string/null）— 前端可直接使用判斷顯示邏輯。
- **AV-002** ✅：`POST /api/auth/register` 的 `referralCode` 為選填欄位，不傳時後端應正常處理 — 已確認。
- **AV-003** ✅：`POST /api/auth/register` 成功後回傳 `token` 和 `refreshToken`，前端可直接完成自動登入 — 已確認。
- **AV-004** ⚠️：推薦碼驗證 API 是否有頻率限制（Rate Limiting）？用戶快速輸入會觸發多次 API，debounce 已緩解，但後端是否有額外保護？
- **AV-005** ⚠️：`POST /api/auth/register` 若 `referralCode` 存在但為無效碼，後端是否拒絕請求（400）或靜默忽略推薦碼繼續建立帳號？前端需知道兩種行為的處理方式。
- **AV-006** ❌：`POST /api/auth/register` Response 缺少 `referralStoreName` 欄位 — 若想在歡迎 Toast 中顯示「您已透過推薦碼加入 {店家}」，需在 Response 中或前端保留驗證結果中的 `storeName`。
- **AV-007** ⚠️：推薦碼是否區分大小寫？前端自動轉大寫是否與後端驗證一致？需後端確認。
- **AV-008** ❌：`POST /api/referral/validate` 未說明推薦碼失效或已使用上限（若有）時的 `message` 錯誤文字 — 建議後端提供標準化的錯誤碼（如 `INVALID_CODE`, `EXPIRED_CODE`）供前端映射特定錯誤訊息。
- **AV-009** ⚠️：`POST /api/auth/register` 成功後，推薦關係是否永久綁定（後端確認）？前端需在確認 Modal 或說明文字中向用戶提示「推薦碼一旦確認後不可更改」。

凡例：✅ 已確認一致 | ⚠️ 需要確認 | ❌ 發現缺口

## 成功標準

- **SC-001**：推薦碼驗證 API 呼叫次數在用戶快速輸入時受 debounce 控制，每次鍵盤輸入不觸發超過 1 次 API 請求（600ms 內）。
- **SC-002**：驗證指示器在 API 回應後 100ms 內更新狀態（⏳ → ✅ 或 ❌）。
- **SC-003**：表單在所有必填欄位有效後，「立即註冊」按鈕才啟用（客戶端驗證，不等 API）。
- **SC-004**：透過推薦連結（`?ref=CODE`）進入頁面後，推薦碼自動填入並驗證，額外操作步驟為零。
- **SC-005**：密碼欄位在任何情況下不以明文形式出現在 URL、localStorage 或 sessionStorage 中。
- **SC-006**：整個推薦碼功能對未使用推薦碼的用戶完全透明（不影響正常註冊流程的步驟數量）。

# Feature Specification: 刮刮樂獎項規則修正與編號指定流程

**Feature Branch**: `015-scratch-lottery-prize-rules`  
**Created**: 2026-04-06  
**Status**: Draft  

## 背景說明

本次修正針對刮刮樂（SCRATCH_CARD）模式的三個核心問題：

1. **後台創建規範不明確**：刮刮樂商品的獎項結構沒有清楚的制度，導致商品配置混亂。
2. **前台玩家指定模式 UI 缺失**：前台在玩家指定模式下，未顯示號碼指定畫面，玩家可直接進行遊戲，違反規則。
3. **編號指定流程不完整**：刮刮樂的籤號分配時機與一番賞/扭蛋不同，尚未正確實作。

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 後台店家創建刮刮樂商品 (Priority: P1)

店家在後台新增刮刮樂商品時，填寫總抽數並設定唯一大獎。系統依此自動確立獎項結構：1 個大獎 + 剩餘全部為「銘謝惠顧」。

**Why this priority**: 這是整個刮刮樂流程的起點，結構不正確則後續流程全部無效。

**Independent Test**: 在後台創建一個 111 抽的刮刮樂商品，確認系統僅允許設定 1 個大獎，且大獎數量鎖定為 1，剩餘 110 個位置自動標示為銘謝惠顧。

**Acceptance Scenarios**:

1. **Given** 店家進入新增刮刮樂表單，**When** 填寫總抽數為 111 並設定大獎等級（如：A 賞 或 刮刮樂專屬賞），**Then** 系統顯示大獎數量固定為 1，且告知剩餘 110 抽皆為銘謝惠顧。
2. **Given** 店家嘗試新增第二個大獎，**When** 提交表單，**Then** 系統拒絕並顯示「刮刮樂模式僅允許一個大獎」。
3. **Given** 商品總抽數為 N，**When** 商品成功創建，**Then** 系統確認大獎數量 = 1，銘謝惠顧數量 = N - 1。

---

### User Story 2 - 前台開套玩家（玩家指定模式）必須先指定大獎號碼 (Priority: P1)

當刮刮樂商品的 `gameMode` 為 `SCRATCH_PLAYER` 時，第一位進入該套的玩家（開套者）在進行任何抽獎前，必須看到「號碼指定畫面」，選擇大獎對應的籤號後才能開始遊戲。

**Why this priority**: 這是玩家指定模式的核心機制，缺少此步驟等於此模式完全失效，且違反玩家對遊戲規則的期待。

**Independent Test**: 創建一個 `SCRATCH_PLAYER` 的刮刮樂商品，以第一個玩家身份進入，確認前台顯示號碼選擇畫面（列出所有可用號碼），未選擇前無法進行抽獎。

**Acceptance Scenarios**:

1. **Given** 商品為 `SCRATCH_PLAYER` 模式且尚未有人指定大獎號碼，**When** 開套玩家進入籤位頁面，**Then** 系統自動顯示「請指定大獎號碼」畫面，列出所有可用的籤號供玩家選擇，並阻擋進入抽獎流程。
2. **Given** 開套玩家在指定畫面選擇了大獎籤號，**When** 確認送出，**Then** 系統記錄指定結果，關閉指定畫面，開放抽獎操作。
3. **Given** 非開套玩家（後來加入的玩家）進入同一套，**When** 大獎號碼尚未被指定，**Then** 系統顯示「等待開套者指定大獎號碼」的提示，並阻擋非開套玩家進行抽獎。
4. **Given** 開套玩家嘗試不經過指定畫面直接呼叫抽獎 API，**When** 送出抽獎請求，**Then** 系統拒絕並回傳 `designationRequired: true` 及可用號碼列表。

---

### User Story 3 - 店家指定模式（SCRATCH_STORE）後台指定大獎號碼 (Priority: P2)

`SCRATCH_STORE` 模式下，店家在後台創建商品後，需指定大獎對應的籤號。完成指定後，系統自動將其餘籤號分配為銘謝惠顧，商品才算設定完成可上架。

**Why this priority**: 店家指定模式是另一條主要流程，確保店家可控制大獎位置。

**Independent Test**: 以店家身份創建 `SCRATCH_STORE` 商品，在後台指定大獎籤號，確認指定後其餘號碼自動補齊，且前台玩家無需任何指定步驟即可直接抽獎。

**Acceptance Scenarios**:

1. **Given** `SCRATCH_STORE` 商品剛創建完成但尚未指定籤號，**When** 店家進入後台指定頁面並選擇大獎籤號，**Then** 系統完成指定並自動將其他籤號分配為銘謝惠顧。
2. **Given** `SCRATCH_STORE` 店家尚未完成籤號指定，**When** 嘗試將商品設為上架，**Then** 系統拒絕並提示「請先完成大獎號碼指定」。（此規則**不適用於** SCRATCH_PLAYER 模式）
3. **Given** `SCRATCH_STORE` 商品已完成籤號指定，**When** 前台玩家進入，**Then** 玩家直接看到籤位選擇畫面，不需要任何指定步驟。

---

### User Story 4 - 籤號分配時機對比（刮刮樂 vs 一番賞/扭蛋）(Priority: P2)

系統需明確區分不同商品類型的籤號分配時機，確保每種模式的資料完整性。

**Why this priority**: 混淆分配時機會導致遊戲邏輯錯誤，影響所有模式的可靠性。

**Independent Test**: 分別創建一番賞、扭蛋、SCRATCH_STORE、SCRATCH_PLAYER 四種商品，確認各自的籤號分配時機符合規則。

**Acceptance Scenarios**:

1. **Given** 創建一番賞或扭蛋商品，**When** 商品儲存完成，**Then** 系統立即為所有籤位分配獎品編號，商品即刻可上架使用。
2. **Given** 創建 `SCRATCH_STORE` 商品，**When** 商品儲存完成，**Then** 籤號尚未分配，需等店家在後台完成大獎號碼指定後，系統才補齊其餘籤號。
3. **Given** 創建 `SCRATCH_PLAYER` 商品，**When** 商品儲存完成，**Then** 籤號尚未分配，商品進入「等待玩家指定」狀態；待前台第一位開套玩家完成大獎號碼指定，系統才補齊其餘籤號。

---

### Edge Cases

- 若玩家指定模式下，開套玩家離開頁面後再返回，且 10 分鐘尚未超時，系統應記住尚未完成指定的狀態，並再次顯示指定畫面。
- 若開套玩家超過 **10 分鐘**未完成指定，系統自動釋放場次，該玩家的開套者身份失效；下一位進入的玩家自動成為新開套者，並看到指定畫面。
- **超時倒數 10 分鐘期間，不允許任何其他玩家搶先指定**；必須等計時器歸零、場次釋放後，新開套者才可指定。此規則循環適用（新開套者超時後再次釋放，以此類推）。
- 若玩家嘗試指定超出可用號碼範圍的籤號，系統應拒絕並提示有效範圍。
- 若店家在 `SCRATCH_STORE` 模式下嘗試指定已被其他流程佔用的籤號，系統應提示衝突並要求重新選擇。
- 若商品總抽數為 1，則大獎數量 = 1，銘謝惠顧數量 = 0（全部為大獎）。
- 若店家在商品上架後嘗試修改已指定的大獎號碼，系統應拒絕（籤號指定後不可更動）。

## Clarifications

### Session 2026-04-06

- Q: SCRATCH_PLAYER 商品若開套者放棄指定，系統如何處理？ → A: 設定 10 分鐘超時，自動釋放場次，下一位進入者成為新開套者；超時倒數期間不允許任何人搶先指定，必須等時間釋放後才由新開套者指定。此規則循環適用。
- Q: 「等待大獎號碼指定」應如何在系統中表達？ → A: Lottery 實體新增獨立的 `designationStatus` 欄位（PENDING / COMPLETED），與現有上架狀態（ON_SHELF / OFF_SHELF）分開管理。
- Q: 刮刮樂的 `revealedNumber` 應何時生成？ → A: 商品創建時立即生成所有 revealedNumber（1~N 的隨機排列），暫不綁定任何獎品；指定流程完成後，才將大獎 revealedNumber 與大獎配對，其餘全部為 null（銘謝惠顧）。
- Q: 大獎號碼指定完成後，商品上架（ON_SHELF）是自動觸發還是需店家手動？ → A: 兩種模式規則不同。SCRATCH_PLAYER：商品須**先上架**，前台玩家進入後才指定號碼（指定發生在遊戲開始時）。SCRATCH_STORE：店家後台先指定號碼，之後**手動自行決定**何時上架。
- Q: 等待指定期間，前台是否應對非開套玩家顯示倒數計時器？ → A: 顯示倒數計時器，讓非開套玩家清楚知道開套者還剩幾分鐘、超時後自己有機會接手成為新開套者。

## Requirements *(mandatory)*

### Functional Requirements

#### 後台商品創建規範

- **FR-001**: 刮刮樂商品創建表單中，店家只需填寫「總抽數」，系統自動計算並顯示：大獎數量 = 1，銘謝惠顧數量 = 總抽數 - 1。
- **FR-002**: 刮刮樂商品的大獎數量必須固定為 1，系統不允許店家新增第二個大獎。
- **FR-003**: 大獎等級選項必須包含完整的 A~Z 等級，並提供一個「刮刮樂大獎」專屬等級選項（例如：GRAND 或 SP 賞），供店家選擇最適合的獎項稱號。
- **FR-004**: 商品成功創建後，系統需在後台商品詳情頁顯示狀態標示：「尚待指定大獎號碼（不可上架）」（SCRATCH_STORE 模式）或「上架後由開套玩家指定大獎號碼」（SCRATCH_PLAYER 模式）。

#### 店家指定模式（SCRATCH_STORE）

- **FR-005**: 後台商品管理頁必須提供「指定大獎號碼」的入口，僅在 `SCRATCH_STORE` 且尚未完成指定的商品上顯示。
- **FR-006**: 店家指定大獎號碼後，系統必須自動將其餘所有籤號分配為銘謝惠顧，整個過程必須在同一交易中完成，確保資料一致性。
- **FR-007**: 未完成大獎號碼指定（`designationStatus = PENDING`）的 **`SCRATCH_STORE` 商品**，系統必須阻擋商品上架操作。`SCRATCH_PLAYER` 商品不受此限制，可在 `designationStatus = PENDING` 的狀態下直接上架。
- **FR-007b**: `SCRATCH_PLAYER` 商品的大獎號碼指定發生在**遊戲中**（前台玩家進入後），而非上架前；因此 `designationStatus = PENDING` 對 `SCRATCH_PLAYER` 而言是正常的上架狀態。

- **FR-019**: 刮刮樂商品（SCRATCH_STORE / SCRATCH_PLAYER）創建時，系統必須自動設定 `designationStatus = PENDING`；一番賞與扭蛋商品不需要此欄位（或預設為 COMPLETED）。
- **FR-020**: 大獎號碼指定完成後，系統必須將 `designationStatus` 更新為 `COMPLETED`，此操作需與籤號補齊（銘謝惠顧分配）在同一交易中完成。
- **FR-021**: 後台商品列表應支援依 `designationStatus` 篩選，讓店家快速找到「尚待指定」的刮刮樂商品。
- **FR-017**: 開套者的指定超時倒數期間（10 分鐘），系統必須拒絕所有其他玩家的指定請求，直到場次釋放後才開放新開套者進行指定。
- **FR-022**: `SCRATCH_PLAYER` 商品的指定等待畫面中，非開套玩家必須看到**倒數計時器**，顯示當前開套者的剩餘指定時間（最多 10 分鐘）；計時器歸零後，頁面自動提示「計時結束，您可以成為新的開套者」並刷新狀態。
- **開套場次（Session）** 新增欄位：`designationDeadline`（指定截止時間，開套者確立後開始計時，10 分鐘後自動失效）。

- **FR-008**: 前台進入 `SCRATCH_PLAYER` 商品的籤位頁時，系統必須先判斷：目前是否有開套者已完成指定。
- **FR-009**: 若尚未完成指定，且當前玩家為開套者，系統必須顯示「大獎號碼指定畫面」，列出所有可用籤號，並阻擋進入抽獎操作。
- **FR-010**: 若尚未完成指定，且當前玩家非開套者，系統必須顯示「等待開套者指定大獎號碼」提示，並阻擋該玩家進行抽獎。
- **FR-011**: 開套玩家完成指定後，系統必須自動補齊其餘籤號（分配為銘謝惠顧），並立即開放所有玩家進行抽獎。
- **FR-012**: 任何玩家嘗試直接呼叫抽獎 API（繞過前台 UI）時，若該商品為 `SCRATCH_PLAYER` 且尚未完成指定，系統必須回傳 `designationRequired: true` 以及可指定的籤號列表。

#### 通用籤號分配規則

- **FR-013**: 一番賞（OFFICIAL_ICHIBAN）與扭蛋（GACHA）商品創建時，系統必須立即分配所有籤號，商品可直接上架。
- **FR-014**: 刮刮樂商品（SCRATCH_STORE / SCRATCH_PLAYER）創建時，籤號暫不分配，需等待指定流程完成。
- **FR-015**: 籤號一旦完成指定（不論 SCRATCH_STORE 或 SCRATCH_PLAYER），該籤號分配不可再更動。

### Key Entities

- **刮刮樂商品（Scratch Lottery）**：商品基本資訊、總抽數、gameMode（SCRATCH_STORE / SCRATCH_PLAYER）、`designationStatus`（PENDING / COMPLETED，僅刮刮樂適用）、上架狀態（ON_SHELF / OFF_SHELF）。
- **大獎（Grand Prize）**：數量固定為 1、等級（A~Z 或專屬賞）、對應的籤號（指定前為空）。
- **籤位（Ticket）**：ticketNumber（物理序號）、revealedNumber（**商品創建時即生成，1~N 隨機排列，指定前不綁定獎品**）、isDesignatedPrize（是否為大獎籤）、status（AVAILABLE / DRAWN / LOCKED）、prizeId（指定完成後大獎籤位填入，其餘維持 null = 銘謝惠顧）。
- **開套場次（Session）**：記錄開套玩家身份、指定是否完成、開套時間、保護期資訊、`designationDeadline`（指定截止時間，開套者確立後 10 分鐘內有效）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 店家在後台創建刮刮樂商品時，表單完整引導店家完成所有必要設定（總抽數、大獎等級），不需要額外操作即可理解獎項結構。
- **SC-002**: `SCRATCH_PLAYER` 模式下，100% 的開套者在進入籤位頁時，看到號碼指定畫面（無法繞過），直到完成指定前無法執行抽獎操作。
- **SC-003**: `SCRATCH_STORE` 模式下，未完成大獎號碼指定的商品，100% 無法被上架，前台玩家無法看到或進入該商品。`SCRATCH_PLAYER` 模式下，商品可在指定完成前上架，指定流程由第一位進入的開套玩家在遊戲中完成。
- **SC-004**: 大獎號碼指定完成後，系統自動補齊所有銘謝惠顧籤號的時間在 3 秒內完成（不論商品總抽數大小）。
- **SC-005**: 一番賞與扭蛋商品的現有抽獎流程不受本次修正影響，100% 通過現有測試案例。
- **SC-006**: `SCRATCH_PLAYER` 模式中，非開套玩家嘗試在指定完成前進行抽獎時，100% 收到清楚的等待提示與倒數計時器，不會誤以為遊戲已開始。
- **SC-007**: `SCRATCH_PLAYER` 模式中，非開套玩家等待指定時，倒數計時器必須即時顯示剩餘秒數，誤差不超過 5 秒；計時器歸零後 5 秒內自動刷新頁面狀態。

## Assumptions

- 刮刮樂模式下銘謝惠顧不視為「獎品」，而是「無獎」的籤位，因此 prizeId = null 代表銘謝惠顧，不需要建立額外的「銘謝惠顧獎品」資料。
- 大獎的數量在整個刮刮樂商品生命週期中固定為 1，不支援多大獎結構。
- 玩家指定的大獎號碼為 `revealedNumber`（顯示號碼），非 `ticketNumber`（物理序號），與現有系統保持一致。
- 店家指定號碼的操作入口位於後台商品管理頁，需要 StoreOwner 或 Admin 權限。
- 「開套者」的判斷邏輯沿用現有 Session 機制（isOpener 欄位）。
